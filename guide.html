1. البنية المعمارية

يتبع المشروع نمط Single Page Application (SPA) خفيف الوزن بدون إطار عمل — HTML/CSS/JavaScript vanilla — مع فصل واضح بين العرض والبيانات والأدوات المساعدة.

crm-platform-mvp/ ├── index.html ← هيكل الصفحة، الشريط الجانبي، شريط العنوان ├── guide.html ← هذا الدليل التقني ├── css/ │ └── styles.css ← مكتبة المكوّنات والثيم (أزرق CRM) └── js/ ├── utils.js ← دوال التنسيق (أرقام، عملة، نسب) ├── mock-api.js ← طبقة بيانات وهمية async └── app.js ← التوجيه، الصفحات، التفاعلات

تدفق البيانات

عند تحميل الصفحة، يُنفَّذ app.js الذي يستدعي دوال CrmAPI من mock-api.js. كل دالة API تُرجع Promise مع تأخير محاكٍ (220–350ms) لمحاكاة الشبكة. النتائج تُحوَّل إلى HTML وتُحقَن في عنصر #view.

التوجيه (Routing)

لا يوجد router خارجي — المتغير route في app.js يحدد الصفحة الحالية. القيم المدعومة:

المسارالصفحةالدالة
dashboardلوحة المتابعةpageDashboard()
leadsالعملاء المحتملونpageLeads()
dealsالصفقات (Kanban/جدول)pageDeals()
contactsجهات الاتصالpageContacts()
tasksالمهامpageTasks()
reportsالتقاريرpageReports()
settingsالإعداداتpageSettings()

مكوّنات الواجهة

  • Shell — .shell يحتوي sidebar + workspace
  • Cards — بطاقات إحصائيات وجداول
  • Kanban — لوحة Kanban لمراحل الصفقات
  • Pipeline — شريط مراحل أفقي
  • Modal — نوافذ تفاصيل الصفقات
  • Toast — إشعارات سريعة

2. التخصيص والعلامة التجارية

الألوان والخط

عدّل متغيرات CSS في :root داخل css/styles.css:

:root {
  --primary: #2563eb;      /* الأزرق الأساسي */
  --primary-hover: #1d4ed8;
  --accent: #f59e0b;       /* الكهرماني */
  --font: "Cairo", system-ui, sans-serif;
}

العلامة التجارية

في index.html:

  • .brand-org — اسم المنتج: «رابط CRM»
  • .brand-sys — الوصف: «منصة إدارة علاقات العملاء SaaS»
  • .brand-seal — أيقونة SVG قابلة للاستبدال
  • #role-select — أدوار المستخدم (مدير، مندوب، مسؤول)

إضافة صفحة جديدة

  1. أضف تعريف الصفحة في كائن PAGES داخل app.js
  2. أنشئ دالة pageXxx() تعرض المحتوى
  3. سجّل الدالة في كائن pages داخل render()
  4. أضف عنصر تنقل في renderNav()
  5. إن لزم، أضف دوال API في mock-api.js
جميع النصوص بالعربية مع dir="rtl". الأرقام تُعرض بالإنجليزية (0–9) عبر class .num ودوال fmt() في utils.js.

3. تحرير البيانات الوهمية

جميع البيانات موجودة في js/mock-api.js داخل IIFE CrmAPI. التعديلات تنعكس فوراً عند إعادة تحميل الصفحة.

المصفوفات الرئيسية

المصفوفةالوصفالحقول الأساسية
leadsعملاء محتملونid, name, company, source, score, status
dealsصفقات B2Bid, title, value (ر.س), stage, probability
contactsجهات اتصالname, company, email, city, sector
tasksمهامtitle, assignee, due, priority, done
teamMembersفريق المبيعاتname, role, quota, achieved
activitiesآخر الأنشطةtext, time, type

حالات العملاء المحتملين

القيم المدعومة في status: new، contacted، qualified، lost

مراحل الصفقات

مُعرَّفة في STAGES وSTAGE_ORDER:

prospect → proposal → negotiation → won
                                    ↘ lost

دالة advanceDeal(id) تنقل الصفقة للمرحلة التالية وترفع الاحتمالية 15%.

مثال: إضافة صفقة جديدة

deals.push({
  id: "DL-8811",
  title: "عقد CRM سنوي",
  company: "شركة مثال",
  contact: "محمد أحمد",
  value: 120000,
  stage: "prospect",
  probability: 20,
  owner: "خالد المنصور",
  closeDate: "2026-04-30",
  products: "Professional",
  notes: "تم التواصل الأولي"
});

4. واجهة Mock API

جميع الدوال async وتُرجع نسخاً من البيانات (أو مراجع معدّلة للتفاعلات).

الدالةالمعاملاتالوصف
getDashboard()—إحصائيات، رسم بياني، أنشطة، ملخص المراحل
getLeads(filter)all | new | contacted | qualified | lostقائمة العملاء المحتملين
qualifyLead(id)معرّف العميلتأهيل عميل (new/contacted → qualified)
getDeals(stage)all | prospect | ... | lostقائمة الصفقات
advanceDeal(id)معرّف الصفقةتقديم مرحلة الصفقة
getContacts(q)نص بحثدليل جهات الاتصال
getTasks(filter)all | open | doneقائمة المهام
toggleTask(id)معرّف المهمةتبديل حالة الإكمال
getReports()—بلاطات التقارير
getSettings()—مراحل، فريق، تكاملات

الانتقال إلى Backend حقيقي

استبدل محتوى mock-api.js بطبقة fetch:

async getDeals(stage = "all") {
  const res = await fetch(`/api/deals?stage=${stage}`, {
    headers: { Authorization: `Bearer ${token}` }
  });
  return res.json();
}

احتفظ بنفس أسماء الدوال وشكل الاستجابة — app.js لن يحتاج تعديلاً كبيراً.

5. النشر والاستضافة

تشغيل محلي

افتح index.html مباشرة في المتصفح، أو استخدم خادم محلي:

# Python
python -m http.server 8080

# Node.js (npx)
npx serve .

# VS Code Live Server
# انقر بزر الماوس الأيمن على index.html → Open with Live Server

ثم افتح: http://localhost:8080

GitHub Pages

  1. أنشئ مستودع GitHub وارفع محتويات المجلد
  2. Settings → Pages → Source: Deploy from branch (main)
  3. Root directory: / أو /crm-platform-mvp
  4. الرابط: https://username.github.io/repo-name/

Netlify / Vercel

اسحب المجلد إلى لوحة Netlify Drop، أو:

# netlify.toml (اختياري)
[build]
  publish = "."
# لا build command — موقع static

متطلبات الاستضافة

  • Static hosting فقط — لا Node.js مطلوب
  • HTTPS موصى به
  • خط Cairo يُحمَّل من Google Fonts (يتطلب اتصال إنترنت)

6. ملاحظات التوسّع

من MVP إلى منتج

  • Backend: Node.js/Express أو .NET أو Laravel مع PostgreSQL — جداول: leads, deals, contacts, tasks, activities, users
  • المصادقة: JWT + RBAC (admin, manager, rep) — selector الأدوار الحالي placeholder
  • Real-time: WebSockets لتحديث Kanban بين أعضاء الفريق
  • التكاملات: Microsoft Graph، SAP، WhatsApp Business API
  • التقارير: تصدير PDF/Excel عبر backend أو jsPDF

تحسينات Frontend

  • استبدال SPA اليدوي بـ React/Vue مع React Router
  • State management (Zustand/Pinia) بدل mutating arrays
  • Virtual scrolling للجداول الكبيرة
  • Drag-and-drop Kanban (SortableJS)
  • i18n لدعم EN/AR

الأمان والامتثال

  • تشفير البيانات الحساسة (PII) في قاعدة البيانات
  • Audit log لكل تغيير على الصفقات
  • الامتثال لنظام حماية البيانات الشخصية السعودي (PDPL)
  • Rate limiting على API

الأداء

  • تجميع CSS/JS في ملف واحد للإنتاج
  • Cache headers للملفات الثابتة
  • CDN لخط Cairo محلياً إن لزم
  • Pagination/server-side filtering للقوائم الطويلة
هذا MVP مصمم للعرض في Portfolio وDemo — البيانات وهمية ولا تُخزَّن بين الجلسات (إلا أثناء الجلسة الحالية في الذاكرة).
